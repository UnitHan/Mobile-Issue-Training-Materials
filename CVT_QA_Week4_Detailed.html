CVT 최종 리포트 — 릴리즈 vX.Y.Z

대상: 모바일 앱 QA 신입 (Week 3 완료자)
목표: "실제 프로젝트처럼" 계획 → 수행 → 결함 처리 → 재검증 → 최종 리포트까지 완주

0) Week 4에서 신입이 반드시 달성해야 하는 역량(Exit Competency)

아래를 스스로 수행할 수 있어야 한다.

1. 릴리즈 변경사항을 기반으로 테스트 범위(스모크/핵심 E2E/영향 범위 회귀)를 구성한다.

2. 리스크 기반으로 우선순위(Go/No-Go 관점)를 제시한다.

3. 결함을 재현 가능한 형태로 보고하고(증거 포함), 트리아지에서 합리적으로 설명한다.

4. Fix 재검증 시 “Fix 확인 + 사이드이펙트 2~3개” 규칙을 적용한다.

5. 최종 CVT 리포트를 작성하고 잔여 리스크를 명확히 기술한다.

1) Week 4 운영 규칙(현장형)

1.1 시간표(권장 90분 기준)

  • 0~10분: 오늘 목표 / 어제 산출물 피드백
  • 10~30분: 이론(원칙 + 예시 2개)
  • 30~80분: 실습(실제 앱/가짜 릴리즈 세트 기반)
  • 80~90분: 산출물 제출/리뷰 포인트 안내

1.2 산출물 제출 규칙(매일)

  • 파일명 예시: `W4D16_TestPlan_v1.md`, `W4D18_TriageSheet.xlsx`, `W4_Final_CVT_Report.md`
  • 모든 산출물은 “한눈에 읽히는 구조(제목/표/항목)”를 사용한다.
  • 결함 리포트는 최소 1개/일(Week4 전체 최소 5개)

1.3 “오늘 반드시 하는 3가지” 공통 체크

  • (1) 스모크 3개 이상 수행
  • (2) 핵심 E2E 1개 이상 수행
  • (3) 결함/리스크 1건 이상 문서화(결함이 없으면 리스크로 기록)

2) Week 4 실습용 시나리오(교육용 릴리즈 패키지)

교육은 실데이터가 없어도 가능해야 하므로, 가상의 릴리즈 노트 + 테스트 계정 + 환경을 표준으로 제공한다.

2.1 가상 릴리즈 노트(예시 템플릿)

  • 변경 A: 로그인 UX 개선(자동로그인, 만료 토큰 처리 개선, 로그인 실패 메시지 변경)
  • 변경 B: 리스트 페이지네이션 개선(성능 최적화, 중복 제거)
  • 변경 C: 딥링크/알림 진입 안정화(비로그인 시 로그인 후 목적지 복귀)
  • 변경 D: 설정 화면 개편(권한 안내/설정 이동 버튼 추가)
  • 서버 변경: 검색 API 파라미터 추가(정렬 옵션)
  • 외부 SDK: 푸시 SDK 버전 업데이트

> 실제 교육에서는 회사 앱/기능에 맞게 변경 항목을 바꿔 쓰면 된다. 구조만 동일하면 됨.

2.2 테스트 계정/데이터 세트(교육용)

  • 계정 A(정상): 가입/로그인 가능, 데이터 적음(리스트 5건)
  • 계정 B(데이터 많음): 리스트 500건, 알림 수신 가능
  • 계정 C(엣지): 특수문자 포함 검색 이력, 설정 토글 다수 변경
  • 결제/구매가 있는 서비스면: “테스트 상품/샌드박스 결제” 계정

2.3 테스트 환경 축(최소 구성)

  • Android 2대(서로 다른 OS) + iOS 2대(서로 다른 OS) 정도를 권장
  • 네트워크: 정상 Wi-Fi + “느림/끊김”을 만들 수 있는 환경(에뮬레이터/네트워크 셰이핑/약한 Wi-Fi 등)
  • 옵션: 다크모드/폰트 크게(접근성)

3) Day 16 — CVT 계획서(Test Plan) 작성 (범위/일정/기기/리스크/종료 기준)

3.1 학습 목표

  • “무엇을, 왜, 어느 정도까지” 볼지 근거를 가진 계획서를 만든다.
  • 계획서가 있으면 테스트가 흔들리지 않는다.

3.2 이론(핵심 개념)

3.2.1 CVT 계획서가 반드시 답해야 하는 7가지 질문

1) 이번 릴리즈 변경은 무엇인가?

2) 영향 범위는 어디까지인가?

3) 우선순위가 높은 리스크는 무엇인가?

4) 어떤 기기/OS/환경에서 할 것인가?

5) 스모크/핵심 E2E/회귀 범위는 무엇인가?

6) 결함 처리(트리아지/재검증) 규칙은 무엇인가?

7) 언제 “통과”로 볼 것인가(Exit Criteria)?

3.2.2 테스트 범위 구성 3단

  • 스모크: 앱이 “사용 가능”한지
  • 핵심 E2E: 사용자가 “목적을 달성”할 수 있는지
  • 영향 범위 회귀: 변경으로 “깨질 수 있는 주변”

3.3 실습(60분)

Step 1) 변경 분석(10분)

  • 릴리즈 노트에서 변경 항목을 4개로 요약
  • 각 변경 항목에 대해 “영향 기능 2개”씩 붙이기

Step 2) 리스크 Top 5 선정(10분)

  • 사용자 영향(매출/핵심 기능/진입불가) 기준
  • 장애 이력/외부 SDK/서버 변경이 있으면 우선순위 상승

Step 3) 테스트 범위 표 작성(20분)

  • 스모크 15개(또는 서비스 상황에 맞춘 최소 10개)
  • 핵심 E2E 5개
  • 영향 범위 회귀 20개(혹은 Top 30 테스트)

Step 4) Exit Criteria 정의(10분)

  • Fatal/Critical 0건
  • 핵심 E2E 100% 통과
  • 잔여 리스크 목록 작성 + 공유 완료

Step 5) 일정/리소스(10분)

  • 누가 어떤 화면/기능을 맡는지 분배
  • 일자별 목표(오늘은 스모크+E2E, 내일은 회귀 등)

3.4 산출물 템플릿(복붙용)

3.4.1 Test Plan (md) 템플릿

  • 제목: CVT Test Plan — 릴리즈 vX.Y.Z
  • 목적:
  • 범위:
  • In Scope:
  • Out of Scope:
  • 변경 요약(릴리즈 노트):
  • 영향 분석(기능/화면/플로우):
  • 리스크 Top 5:
  • 테스트 전략:
  • Smoke:
  • Core E2E:
  • Regression:
  • 환경/기기:
  • 테스트 데이터/계정:
  • 결함 처리 규칙:
  • Severity / Priority 기준(조직 기준)
  • 재검증 규칙(사이드이펙트 2~3개)
  • Exit Criteria:
  • 일정/담당:

3.5 평가 루브릭(채점 기준)

  • (A) 변경/영향 분석이 구체적이다(화면/플로우 단위)
  • (B) 리스크 우선순위에 근거가 있다
  • (C) 범위가 과도하지 않다(시간 대비 현실적)
  • (D) Exit Criteria가 판단 가능하다(측정/조건 명확)

4) Day 17 — 스모크 + 핵심 E2E 구성 (Go/No-Go 관점)

4.1 학습 목표

  • “릴리즈 판단”에 가장 필요한 테스트 세트를 정의하고 실행한다.

4.2 이론(핵심 개념)

4.2.1 스모크는 “실패하면 바로 No-Go” 항목

예)

  • 앱 실행/로그인/메인 진입
  • 핵심 탭 전환/리스트 표시
  • 핵심 액션 1개(저장/구매/통화/재생 등)

4.2.2 핵심 E2E는 “사용자 목적 달성” 단위

예)

  • 알림 → 상세 → 액션 완료
  • 검색 → 필터/정렬 → 상세 → 완료
  • 로그인 → 설정 변경 → 정상 반영

4.3 실습(70분)

Step 1) 스모크 15개 확정(10분)

  • “실패하면 사용자에게 어떤 피해?”를 각 항목에 1줄씩 적기(리스크 연결)

Step 2) 핵심 E2E 5개 확정(10분)

  • “목적 달성”이 명확한 5개 선택(최소 3개는 ‘변경된 영역’ 포함)

Step 3) 실행 및 결과 기록(40분)

  • 스모크 수행(기기 2종 이상)
  • 핵심 E2E 2개 이상 수행(최소 1개는 알림/딥링크/비정상 진입 포함)

Step 4) 실패/리스크 기록(10분)

  • 결함이 없으면 리스크로 기록(“잠재 결함 가설” + 재검증 필요)

4.4 산출물

  • `W4D17_SmokeChecklist.md` (체크 결과 포함)
  • `W4D17_CoreE2E.md` (E2E 정의 + 수행 결과)

4.5 스모크 체크리스트 템플릿(복붙용)

  • [ ] 앱 설치/업데이트
  • [ ] 앱 실행(콜드 스타트)
  • [ ] 로그인(또는 비로그인 진입)
  • [ ] 메인 진입(핵심 탭/홈)
  • [ ] 리스트 표시(데이터 로딩)
  • [ ] 상세 진입(아이템 1개)
  • [ ] 핵심 액션 1개 수행(서비스에 맞춤)
  • [ ] 뒤로/닫기 동작(스택 정상)
  • [ ] 네트워크 실패 시 에러 처리(최소 1회)
  • [ ] 종료/재실행(기본 상태 확인)

4.6 평가 루브릭

  • (A) “Go/No-Go” 관점으로 항목 선정이 합리적
  • (B) 결과 기록이 명확(기기/OS/버전 포함)
  • (C) 실패 시 영향과 결함/리스크 분리가 명확

5) Day 18 — 결함 트리아지/우선순위/커뮤니케이션 훈련

5.1 학습 목표

  • 결함을 “기술적으로 재현 가능”하게 설명하고, 트리아지에서 합리적으로 우선순위를 정한다.

5.2 이론(핵심 개념)

5.2.1 Severity(심각도) vs Priority(우선순위)

  • Severity: 결함 자체의 피해 크기(크래시/데이터 손상/결제 불가 등)
  • Priority: “언제 고쳐야 하는가”(릴리즈 차단 여부)

5.2.2 트리아지에서 자주 나오는 상황과 대응

  • 재현 불가: 환경/조건 명시 + 로그/영상 + 재현률(간헐/조건부) 기록
  • 기획 이슈: 기대 결과 근거(스펙/정책) 명시
  • 중복 이슈: 원인/증상/재현이 같은지 비교 후 통합

5.3 실습(75분)

Step 1) 결함 5개 이상 확보(또는 시뮬레이션 카드 제공)(15분)

  • 실제 결함이 없으면 “결함 카드(가상 이슈)”로 트리아지 훈련 가능

Step 2) 결함 10개 트리아지 표 작성(20분)

필드(권장)

  • ID / 제목 / 재현률 / 사용자 영향 / 심각도 / 우선순위 / 담당 / 릴리즈 차단 여부 / 근거 / 상태

Step 3) 트리아지 회의 롤플레잉(30분)

역할

  • QA(신입): 결함 설명
  • Dev: 재현/원인 질문
  • PO/PM: 출시 영향 질문
  • Lead: 우선순위 결정

Step 4) 결과 반영(10분)

  • 우선순위/담당/조치 계획 업데이트

5.4 산출물

  • `W4D18_TriageSheet.md` 또는 xlsx
  • 결함 리포트 최소 1개(실제 또는 시뮬레이션)

5.5 결함 리포트 템플릿(복붙용)

제목: [Severity][영역] 한 줄 요약(무엇이/어떻게)

  • 환경: OS/기기/앱버전/네트워크/계정상태
  • 재현 절차(5~8단계):

1)

2)

3)

  • 실제 결과(Actual):
  • 기대 결과(Expected):
  • 빈도: 100% / 간헐(조건)
  • 영향: 사용자 피해(진입불가/중복처리/데이터오염 등)
  • 증거: 스크린샷/영상/로그(가능한 범위)
  • 비고: 우회 여부/제한사항

5.6 평가 루브릭

  • (A) 재현 절차가 다른 사람이 그대로 따라할 수 있다
  • (B) Expected 근거가 있다(정책/플로우/화면 문구 등)
  • (C) 사용자 영향 설명이 명확하다
  • (D) 트리아지에서 우선순위 근거 제시 가능

6) Day 19 — 재검증 전략(사이드이펙트 포함) + 리그레션 운영

6.1 학습 목표

  • Fix 확인만 하고 끝내지 않고, 주변 영향(사이드이펙트)을 체계적으로 잡는다.

6.2 이론(핵심 개념)

6.2.1 재검증 기본 규칙(현장 표준)

  • Fix 확인 1회
  • 사이드이펙트 2~3개(같은 화면의 다른 기능 / 반대 방향 흐름 / 인접 기능) 확인
  • 관련 자동화가 있으면 “최소 스모크” 재실행

6.2.2 사이드이펙트 선정 3가지 패턴

  • 인접 기능: 같은 화면의 다른 버튼/옵션/필터
  • 반대 방향 흐름: 추가가 고쳐졌으면 삭제/취소 흐름 확인
  • 경유 흐름: 로그인/권한/딥링크 같은 경유가 있는지 확인

7) Day 20 — 최종 리포트/릴리즈 판단/잔여 리스크 문서화

7.1 학습 목표

  • “결함이 몇 개”가 아니라, “릴리즈가 안전한지”를 의사결정할 수 있게 리포트를 만든다.

7.2 이론(핵심 개념)

7.2.1 최종 리포트에서 반드시 답해야 하는 6가지

1) 이번 릴리즈에서 테스트 범위는 무엇이었나?

2) 스모크/핵심 E2E는 통과했나?

3) 차단 결함(Fatal/Critical)은 0건인가?

4) 남은 결함/리스크는 무엇이며 영향은?

5) 우회/제한/모니터링 계획은?

6) 최종 판단(Go/No-Go/Conditional Go)과 근거는?

7.2.2 “잔여 리스크”를 잘 쓰는 방법

  • 리스크는 결함이 아니어도 된다.
  • 문장 공식
  • 조건(언제) + 현상(무엇이) + 영향(왜 위험) + 대응(어떻게 완화)

예)

  • 조건: 네트워크 느림에서 로그인 직후
  • 현상: 홈 로딩이 5~7초 지연
  • 영향: 사용자 이탈 가능
  • 대응: 모니터링 + 성능 로그 확인 + 다음 핫픽스 후보

7.3 실습(90분)

Step 1) 범위/결과 요약(20분)

Step 3) 잔여 리스크 3~5개 작성(20분)

실제 결함이 없을 때도 트리아지/재검증 훈련이 가능하도록 제공한다.

1) 로그인 후 목적지 복귀 실패(알림 딥링크 진입)

3) 설정에서 권한 변경 후 앱 복귀 시 토글 상태 미갱신

4) 로딩 중 저장 버튼이 활성화되어 중복 요청 발생

5) 검색어에 특수문자 입력 시 결과 0건(인코딩 문제 의심)

6) 네트워크 끊김 후 재시도 버튼이 동작하지 않음

7) iOS에서 스와이프 백 제스처로 화면이 중간에 멈춤(gesture 충돌)

8) Android에서 뒤로가기로 모달이 닫히지 않고 앱이 종료됨

9) 콜드 스타트에서 첫 화면 진입이 8초 이상(Watchdog/ANR 리스크)

10) 로그아웃 후 이전 사용자 데이터 일부가 남아 있음(정합성/캐시)
