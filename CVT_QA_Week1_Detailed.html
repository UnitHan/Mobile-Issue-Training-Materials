0) Week 1에서 신입이 반드시 달성해야 하는 역량(Exit Competency)

아래를 스스로 수행할 수 있어야 한다.

1. 모바일 앱과 웹의 차이점을 설명하고 모바일 특화 테스트 포인트를 나열한다.

4. 테스트 케이스를 읽고 수행 결과를 기록할 수 있다.

5. 간단한 결함을 발견하고 스크린샷을 첨부할 수 있다.

1) Week 1 운영 규칙(현장형)

1.1 시간표(권장 90분 기준)

  • 0~10분: 오늘 목표 및 개요 설명
  • 10~35분: 이론(개념 + 예시)
  • 35~80분: 실습(도구 사용 또는 간단한 테스트)
  • 80~90분: 오늘 배운 내용 정리 및 질의응답

1.2 산출물 제출 규칙(매일)

  • 파일명 예시: `W1D1_Summary.md`, `W1D3_FirstTestCase.md`
  • 모든 산출물은 간단한 메모 형태로 작성(3~5개 항목)

1.3 "오늘 반드시 하는 3가지" 공통 체크

  • (2) 실습 도구 1개 이상 직접 사용
  • (3) 질문 1개 이상 작성(이해 안 되는 부분 기록)

2) Day 1 — 모바일 앱 QA 소개 및 환경 구성

2.1 학습 목표

  • 모바일 앱 QA가 무엇인지 이해한다.
  • Android/iOS 개발자 도구를 설치하고 기기를 연결한다.

2.2 이론(핵심 개념)

2.2.1 모바일 앱 vs 웹 차이점

구분 웹 모바일 앱
실행 환경 브라우저 OS (Android/iOS)
설치 불필요 필요 (스토어)
오프라인 제한적 가능
권한 브라우저 권한 OS 권한 (카메라/위치/알림 등)
업데이트 자동 (새로고침) 수동 (스토어 업데이트)
디바이스 접근 제한적 센서/카메라/GPS 등 직접 접근

2.2.2 모바일 앱 테스트 특화 포인트

  • 다양한 OS 버전/기기 조합
  • 네트워크 상태 변화 (Wi-Fi ↔ 모바일 데이터 ↔ 오프라인)
  • 권한 허용/거부 시나리오
  • 배터리/메모리 제약
  • 화면 크기/해상도 다양성
  • 앱 라이프사이클 (포그라운드 ↔ 백그라운드 ↔ 종료)

2.2.3 CVT(Customer Verification Test)란?

  • 정의: 실제 사용자 관점에서 앱이 정상 동작하는지 검증하는 최종 테스트
  • 시점: 개발 완료 후, 스토어 출시 전
  • 목적: 릴리즈 Go/No-Go 판단

2.3 실습(60분)

Step 1) 개발자 도구 설치 (30분)

Android:
  • ADB (Android Debug Bridge) 설치
  • USB 디버깅 활성화
  • 기기 연결 확인: `adb devices`
iOS:
  • Xcode 설치 (Mac 필요)
  • 기기 연결 확인

Step 2) 첫 앱 설치 및 실행 (20분)

  • 테스트용 앱 APK/IPA 설치
  • 앱 실행 후 메인 화면 캡처
  • 앱 버전 정보 확인

Step 3) 로그 확인 연습 (10분)

  • Android: `adb logcat`
  • iOS: Xcode Console

2.4 산출물

  • `W1D1_Summary.md`: 오늘 배운 핵심 개념 5개 정리
  • 환경 구성 완료 증빙 (기기 연결 스크린샷)

2.5 평가 루브릭

  • (A) 모바일/웹 차이점 3개 이상 설명 가능
  • (B) 기기 연결 및 로그 확인 성공
  • (C) 앱 설치 및 버전 확인 완료

3) Day 2 — UI 부품 분류 및 화면 구성 이해

3.1 학습 목표

  • 앱 화면을 "테스트 가능한 부품"으로 분해할 수 있다.
  • 각 부품별 테스트 포인트를 나열할 수 있다.

3.2 이론(핵심 개념)

3.2.1 UI 부품 6가지 분류

1. 네비게이션: 상단바, 하단 탭, 뒤로가기, 닫기

2. 리스트/피드: 무한 스크롤, Pull-to-refresh, 아이템

3. 입력/폼: TextField, 검색, 피커, 체크박스, 토글

4. 버튼/액션: Primary, Secondary, 아이콘 버튼

5. 팝업/오버레이: Alert, 바텀시트, 토스트, 모달

6. 상태 표현: 로딩, 정상, 빈 화면, 에러

3.2.2 4상태 모델 (모든 화면에 적용)

  • Loading: 데이터 로딩 중 (스피너/스켈레톤)
  • OK: 정상 데이터 표시
  • Empty: 데이터 없음 (빈 화면 안내)
  • Error: 에러 발생 (재시도 버튼)

3.3 실습(70분)

Step 1) 앱 화면 3개 선택 (10분)

  • 로그인 화면
  • 메인/홈 화면
  • 리스트 화면

Step 2) 각 화면의 UI 부품 분류 (30분)

  • 6가지 분류로 표 작성
  • 각 부품에 대한 테스트 포인트 1~2개 작성

예시:

화면 부품 종류 구체적 요소 테스트 포인트
로그인 입력/폼 이메일 입력 형식 검증, 에러 메시지
로그인 버튼 로그인 버튼 비활성화 조건, 로딩 상태
메인 네비게이션 하단 탭 탭 전환, 상태 유지

Step 3) 4상태 확인 (20분)

  • 선택한 화면에서 Loading/OK/Empty/Error 상태 재현
  • 각 상태 스크린샷 캡처

Step 4) 정리 (10분)

  • 부품별 테스트 포인트 요약

2.4 산출물

  • `W1D2_UI_Components.md`: 화면별 부품 분류 표
  • 4상태 스크린샷 (최소 3개 화면 × 4상태 = 12장)

3.5 평가 루브릭

  • (A) UI 부품 6가지로 정확히 분류
  • (B) 4상태 중 최소 3개 상태 확인 완료
  • (C) 테스트 포인트가 구체적이고 실행 가능

4) Day 3 — 테스트 케이스 읽기 및 수행

4.1 학습 목표

  • 테스트 케이스의 구조를 이해한다.
  • 주어진 테스트 케이스를 수행하고 결과를 기록한다.

4.2 이론(핵심 개념)

4.2.1 테스트 케이스 구조

제목: 로그인 - 정상 시나리오

사전 조건:


    

  • 앱 설치 완료
    • 

  • 유효한 계정 준비 (test@example.com / password123)
    • 


  

절차:


1. 앱 실행


2. 로그인 화면에서 이메일 입력: test@example.com


3. 비밀번호 입력: password123


4. "로그인" 버튼 클릭

  

기대 결과:


    

  • 메인 화면으로 이동
    • 

  • 상단에 사용자 이름 표시
    • 

  • 로그인 상태 유지 (재실행 시에도 로그인 상태)
    •

4.2.2 테스트 결과 기록 방법

  • Pass: 기대 결과와 일치
  • Fail: 기대 결과와 불일치 (결함 발견)
  • Blocked: 사전 조건 미충족으로 수행 불가
  • Skip: 시간/환경 부족으로 미수행

4.3 실습(70분)

Step 1) 샘플 테스트 케이스 10개 제공 (10분)

  • 로그인/로그아웃
  • 리스트 조회
  • 검색
  • 설정 변경

Step 2) 테스트 케이스 수행 (50분)

  • 최소 5개 케이스 수행
  • 각 케이스별 결과 기록 (Pass/Fail/Blocked/Skip)
  • Fail 시 스크린샷 캡처

Step 3) 결과 요약 (10분)

  • 수행 결과 통계: Pass N개 / Fail N개
  • 발견된 이슈 간단 요약

4.4 산출물

  • `W1D3_TestExecution.md`: 테스트 수행 결과표
  • 결함 스크린샷 (Fail인 경우)

5) Day 4 — 결함 보고 기초

5.1 학습 목표

  • 결함을 발견했을 때 필수 정보를 기록할 수 있다.
  • 간단한 결함 보고서를 작성할 수 있다.

5.2 이론(핵심 개념)

5.2.1 결함 보고서 필수 항목

1. 제목: 한 줄로 무엇이 문제인지

2. 환경: OS, 기기, 앱 버전

3. 재현 절차: 1, 2, 3 단계로 명확히

4. 실제 결과: 무엇이 잘못되었는지

5. 기대 결과: 어떻게 되어야 하는지

6. 증거: 스크린샷 또는 영상

5.2.2 좋은 결함 보고서 vs 나쁜 결함 보고서

나쁜 예: 
제목: 로그인 안 됨

내용: 로그인이 안 됩니다.
좋은 예: 
제목: [로그인] 잘못된 비밀번호 입력 시 에러 메시지 미표시

환경: Android 12 / Galaxy S21 / 앱 v1.0.0


재현 절차:


1. 앱 실행


2. 이메일 입력: test@example.com


3. 비밀번호 입력: wrongpassword


4. 로그인 버튼 클릭


실제 결과: 아무 반응 없음


기대 결과: "비밀번호가 올바르지 않습니다" 에러 메시지 표시


증거: [스크린샷 첨부]
