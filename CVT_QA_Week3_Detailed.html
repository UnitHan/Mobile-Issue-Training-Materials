Week 3 결함 보고서 포트폴리오

대상: 모바일 앱 QA 신입 (Week 2 완료자)
목표: 고품질 결함 보고서 10개 완성

0) Week 3에서 신입이 반드시 달성해야 하는 역량(Exit Competency)

아래를 스스로 수행할 수 있어야 한다.

1. 결함을 체계적으로 발견할 수 있다 (탐색적 테스팅 기법 적용).

2. 결함을 재현 가능한 형태로 보고할 수 있다 (재현율 100%).

3. Severity(심각도)와 Priority(우선순위)를 구분하고 설정할 수 있다.

4. 트리아지 회의에서 결함을 효과적으로 설명할 수 있다.

5. 회귀 테스트 범위를 선정하고 실행할 수 있다.

1) Week 3 운영 규칙(현장형)

1.1 시간표(권장 90분 기준)

  • 0~10분: 오늘 목표 및 Week 2 복습
  • 10~25분: 이론(결함 발견 기법 + 보고 원칙)
  • 25~80분: 실습(탐색적 테스팅 또는 결함 보고)
  • 80~90분: 산출물 리뷰 및 개선 사항 논의

1.2 산출물 제출 규칙(매일)

  • 파일명 예시: `W3D11_BugReport_Login.md`, `W3D13_TriageNotes.md`
  • 결함 보고서는 "재현 절차 + 증거" 필수
  • 트리아지 회의록 작성 (결정 사항 기록)

1.3 "오늘 반드시 하는 3가지" 공통 체크

  • (1) 결함 최소 3개 이상 발견 및 보고
  • (2) 재현율 100% 달성 (또는 조건부 재현 명시)
  • (3) Severity/Priority 설정 및 근거 작성

2) Day 11 — 탐색적 테스팅 (Exploratory Testing)

2.1 학습 목표

  • 테스트 케이스 없이도 결함을 발견할 수 있다.
  • 탐색적 테스팅 기법을 적용할 수 있다.

2.2 이론(핵심 개념)

2.2.1 탐색적 테스팅이란?

  • 정의: 테스트 설계와 수행을 동시에 진행하며, 앱을 탐색하면서 결함을 발견하는 기법
  • 특징:
  • 스크립트(테스트 케이스) 없음
  • 테스터의 직관과 경험 활용
  • 빠른 피드백

2.2.2 탐색적 테스팅 5가지 기법

1. 경계값 테스팅: 최소/최대/0/음수 입력

2. 데이터 변형: 특수문자/이모지/매우 긴 텍스트

3. 빠른 조작: 연속 클릭/빠른 화면 전환

4. 중단 시나리오: 로딩 중 뒤로가기/앱 전환

5. 조합 탐색: 여러 기능을 빠르게 조합

2.2.3 탐색적 테스팅 차터(Charter) 작성

차터: 로그인 화면 보안 및 입력 검증 탐색

시간: 30분


영역: 로그인 화면


목표:


    

  • 입력 필드 검증 확인
    • 

  • 에러 메시지 적절성
    • 

  • 비정상 입력 처리
    •

2.3 실습(75분)

Step 1) 탐색적 테스팅 차터 3개 작성 (15분)

예시:

  • 차터 1: 로그인 화면 입력 검증
  • 차터 2: 리스트 화면 스크롤 및 필터
  • 차터 3: 설정 화면 권한 및 토글

Step 2) 차터별 탐색 수행 (45분)

  • 각 차터당 15분 타임박스
  • 발견한 결함 즉시 메모

Step 3) 결함 정리 및 우선순위 설정 (15분)

  • 발견한 결함 목록 작성 (최소 5개)
  • Severity 설정

2.4 산출물

  • `W3D11_ExpTest_Charter.md`: 탐색적 테스팅 차터 3개
  • `W3D11_BugList.md`: 발견된 결함 목록 (최소 5개)

  • (B) 최소 5개 이상 결함 발견
  • (C) 탐색 시간을 준수 (타임박스)
  • (D) 다양한 테스팅 기법 적용

3) Day 12 — 재현 가능한 결함 보고서 작성

3.1 학습 목표

  • 결함을 "다른 사람이 재현 가능"한 형태로 보고할 수 있다.
  • 필수 정보(환경/절차/증거)를 누락 없이 작성한다.

3.2 이론(핵심 개념)

3.2.1 재현 가능한 보고서 = 좋은 보고서

  • 개발자가 즉시 재현하여 수정 가능
  • 트리아지에서 우선순위 판단 가능
  • QA가 재검증 가능

3.2.2 결함 보고서 필수 7가지 항목

1. 제목: [Severity][영역] 한 줄 요약

2. 환경: OS/기기/앱버전/네트워크/계정

3. 재현 절차: 5~8단계 (구체적)

4. 실제 결과: 무엇이 잘못되었는지

5. 기대 결과: 어떻게 되어야 하는지 (근거 포함)

6. 빈도: 100% / 간헐(조건 명시)

7. 증거: 스크린샷/영상/로그

3.2.3 재현 절차 작성 TIP

  • 구체적 값 사용: "텍스트 입력" (X) → "test@example.com 입력" (O)
  • 단계 번호 사용: 1, 2, 3...
  • 조건 명시: "로그인 안 된 상태에서"
  • 타이밍 명시: "로딩 완료 전에"

3.3 실습(75분)

Step 1) Day 11에서 발견한 결함 5개 선정 (10분)

Step 2) 각 결함을 재현 가능한 형태로 작성 (50분)

  • 결함당 10분 할당
  • 필수 7가지 항목 모두 포함
  • 스크린샷 첨부

Step 3) 동료 리뷰 (15분)

  • 다른 사람이 재현 절차만 보고 재현 가능한지 확인
  • 피드백 반영

3.4 산출물

  • `W3D12_BugReport_001.md` ~ `W3D12_BugReport_005.md`: 결함 보고서 5개

3.5 결함 보고서 템플릿

```markdown

환경

  • OS: Android 12
  • 기기: Samsung Galaxy S21
  • 앱 버전: v1.2.0
  • 네트워크: Wi-Fi (정상)
  • 계정: 비로그인 상태

재현 절차

1. 앱 실행

2. 로그인 화면에서 이메일 입력: test@example.com

3. 비밀번호 입력: wrongpassword (의도적으로 잘못된 비밀번호)

4. "로그인" 버튼 클릭

5. 결과 확인

실제 결과

  • 화면 변화 없음
  • 에러 메시지 미표시
  • 로그인 버튼 다시 활성화됨

기대 결과

  • "이메일 또는 비밀번호가 올바르지 않습니다" 에러 메시지 표시
  • 비밀번호 입력 필드 하단에 빨간색으로 표시

빈도

  • 재현율: 100% (5/5회 재현)

영향

  • 사용자가 로그인 실패 이유를 알 수 없음
  • 반복 시도로 인한 계정 잠금 위험

증거

![로그인 실패 화면](screenshot_login_fail.png)

추가 정보

  • 네트워크 로그 확인 결과: 401 Unauthorized 응답 수신됨
  • 서버는 정상 응답하나, 클라이언트에서 에러 처리 누락 추정

```

3.6 평가 루브릭

  • (A) 재현 절차를 따라하면 100% 재현 가능
  • (B) 환경 정보 완전 (OS/기기/버전/네트워크)
  • (C) 기대 결과에 근거 명시
  • (D) 증거(스크린샷) 첨부
  • (E) 재현율 명시

4) Day 13 — Severity/Priority 및 트리아지

4.1 학습 목표

  • Severity와 Priority를 구분하고 설정할 수 있다.
  • 트리아지 회의에서 결함을 효과적으로 설명할 수 있다.

4.2 이론(핵심 개념)

4.2.1 Severity(심각도) vs Priority(우선순위)

구분 Severity (심각도) Priority (우선순위)
정의 결함 자체의 피해 크기 언제 고쳐야 하는가
기준 기능 영향/사용자 피해 릴리즈 일정/비즈니스
예시 Critical: 크래시 P0: 릴리즈 차단

4.2.2 Severity 분류 기준

  • Critical: 앱 사용 불가 (크래시/진입불가/결제불가)
  • Major: 주요 기능 동작 안 함 (로그인 실패/데이터 손실)
  • Minor: 사소한 불편 (UI 깨짐/오타)

4.2.3 트리아지 회의 구성

  • 참석자: QA, 개발자, PM/PO, Lead
  • 목적:
  • 결함 재현 확인
  • 우선순위 결정
  • 담당자 할당
  • 릴리즈 차단 여부 판단

4.3 실습(75분)

Step 1) 결함 10개 Severity 설정 (15분)

  • Day 11~12에서 발견한 결함 사용
  • 각 결함에 Severity 부여 및 근거 작성

Step 2) 트리아지 회의 시뮬레이션 (40분)

역할:
  • QA: 결함 설명 (재현 절차/영향)
  • Dev: 재현/원인 질문
  • PM: 출시 영향 질문
  • Lead: 우선순위 결정
절차:

1. QA가 결함 설명 (3분)

2. Dev 질의응답 (2분)

3. PM 비즈니스 영향 검토 (2분)

4. Lead 우선순위 결정 (1분)

5. 다음 결함으로 이동

Step 3) 트리아지 결과 정리 (20분)

  • 결함별 Severity/Priority/담당자/릴리즈 차단 여부 표 작성

3.4 산출물

  • `W3D13_SeverityAnalysis.md`: 결함별 Severity 분석
  • `W3D13_TriageSheet.md`: 트리아지 회의 결과표

4.6 평가 루브릭

  • (A) Severity 설정 근거가 합리적
  • (B) 트리아지 설명이 명확하고 간결 (3분 이내)
  • (C) 개발자 질문에 즉시 답변 가능
  • (D) 우선순위 판단이 비즈니스 영향 고려

5) Day 14 — 회귀 테스트(Regression Test) 설계

5.1 학습 목표

  • 회귀 테스트 범위를 선정할 수 있다.
  • Fix 검증 및 사이드이펙트를 확인할 수 있다.

5.2 이론(핵심 개념)

5.2.1 회귀 테스트란?

  • 정의: 변경(수정/신규 기능)으로 인해 기존 기능이 깨지지 않았는지 확인하는 테스트
  • 시점: Fix 후, 릴리즈 전

5.2.2 회귀 테스트 범위 선정 3단계

1. 직접 영향: Fix된 기능 자체

2. 간접 영향: Fix와 데이터/화면을 공유하는 기능

3. 핵심 기능: 스모크 테스트 (항상 수행)

5.2.3 사이드이펙트 체크 규칙

  • Fix 확인: 원래 결함이 해결되었는지
  • 사이드이펙트 2~3개:
  • 같은 화면의 다른 버튼
  • 반대 방향 흐름 (추가 Fix → 삭제 확인)
  • 인접 기능

5.3 실습(75분)

Step 1) Fix 시나리오 3개 가정 (15분)

예시:

  • Fix 1: 로그인 에러 메시지 표시 수정
  • Fix 2: 리스트 페이지네이션 중복 제거
  • Fix 3: 설정 토글 동기화 수정

Step 2) 각 Fix별 회귀 테스트 범위 선정 (20분)

Fix 직접 영향 간접 영향 사이드이펙트
로그인 에러 로그인 실패 시나리오 회원가입, 비밀번호 찾기 로그아웃 → 재로그인

Step 3) 회귀 테스트 수행 (30분)

  • 선정한 범위 테스트 수행
  • 결과 기록

Step 4) 회귀 테스트 체크리스트 작성 (10분)

  • 향후 반복 사용 가능한 체크리스트

5.4 산출물

  • `W3D14_RegressionPlan.md`: 회귀 테스트 범위 선정
  • `W3D14_RegressionChecklist.md`: 회귀 테스트 체크리스트

5.5 회귀 테스트 체크리스트 템플릿

```markdown

Fix 내용

로그인 실패 시 에러 메시지 표시 추가

직접 영향 범위

  • [ ] 잘못된 비밀번호 입력 시 에러 메시지 표시
  • [ ] 존재하지 않는 이메일 입력 시 에러 메시지 표시
  • [ ] 네트워크 오류 시 에러 메시지 표시

간접 영향 범위
```
