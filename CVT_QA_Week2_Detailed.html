대상: 모바일 앱 QA 신입 (Week 1 완료자)
목표: 화면을 체계적으로 분해하고, 조합 테스트를 설계할 수 있다.

0) Week 2에서 신입이 반드시 달성해야 하는 역량(Exit Competency)

아래를 스스로 수행할 수 있어야 한다.

1. 화면 간 전환(라우팅) 및 데이터 흐름을 파악할 수 있다.

2. 조합 테스트 축(사용자 상태 × 네트워크 × 권한)을 선정하고 시나리오를 만든다.

3. E2E(End-to-End) 시나리오를 작성하고 수행할 수 있다.

4. 결함 가설을 "조건 + 행동 + 기대 + 위험" 공식으로 작성한다.

5. 테스트 설계 문서(Test Design)를 작성할 수 있다.

2) Day 6 — 화면 간 전환 및 라우팅 이해

2.1 학습 목표

  • 앱의 화면 흐름을 그릴 수 있다.
  • 라우팅(딥링크/알림/외부 앱) 진입 경로를 파악한다.

2.2 이론(핵심 개념)

2.2.1 화면 전환의 3가지 패턴

1. 일반 전환: A → B → C (탭/버튼 클릭)

2. 딥링크 진입: 알림/링크 클릭 → 특정 화면 직접 진입

3. 외부 앱 복귀: 앱 전환 (카메라/갤러리/결제) 후 복귀

2.2.2 라우팅 결함 패턴

  • 로그인 후 목적지 복귀 실패 (딥링크)
  • 스택 꼬임 (뒤로가기 시 이전 화면 없음)
  • 탭 전환 후 스크롤/필터 상태 초기화

2.2.3 화면 흐름도(Flow Map) 작성 규칙

[홈] → [리스트] → [상세]

↓         ↓


[설정]    [검색]


↓


[로그아웃] → [로그인]

Step 2) 화면 흐름도 작성 (30분)

Step 3) 라우팅 테스트 포인트 도출 (20분)

  • 각 경로에서 테스트해야 할 항목 작성

예시:

경로 테스트 포인트

상세 → 카메라 → 복귀 상세 화면 데이터 유지
탭1 → 탭2 → 탭1 스크롤 위치 유지

Step 4) 정리 (10분)

3) Day 7 — 데이터 흐름 및 상태 동기화

3.1 학습 목표

  • 화면 간 데이터 전달 및 상태 동기화를 이해한다.
  • 데이터 정합성 결함을 발견할 수 있다.

3.2 이론(핵심 개념)

3.2.1 데이터 흐름 패턴

1. 단방향: A → B (A에서 B로 데이터 전달)

2. 양방향: A ↔ B (B에서 수정 후 A 갱신)

3. 글로벌 상태: 모든 화면에서 공유 (로그인 정보, 설정)

3.2.2 데이터 정합성 결함 패턴

  • 화면 A에서 삭제했는데 화면 B에 여전히 표시
  • 설정 토글 변경 후 다른 화면에서 미반영
  • 캐시와 서버 데이터 불일치

3.2.3 상태 동기화 체크 규칙

  • 데이터 변경 → 관련 화면 모두 확인
  • 로그아웃 → 모든 개인 정보 초기화 확인
  • 설정 변경 → 앱 전체에 반영 확인

3.3 실습(70분)

Step 1) 데이터 변경 시나리오 5개 선정 (15분)

예시:

  • 리스트에서 아이템 삭제 → 상세 화면 확인
  • 설정에서 다크모드 변경 → 모든 화면 확인
  • 프로필 수정 → 홈/마이페이지 확인

Step 2) 각 시나리오 수행 (40분)

  • 데이터 변경 후 관련 화면 3개 이상 확인
  • 동기화 실패 시 스크린샷 캡처

Step 3) 데이터 정합성 체크리스트 작성 (15분)

변경 항목 영향 화면 확인 결과 이슈
아이템 삭제 리스트, 상세, 카트 Pass -
다크모드 ON 모든 화면 Fail 설정 화면만 반영됨

3.5 평가 루브릭

4) Day 8 — 조합 테스트 설계 (3축 조합)

4.1 학습 목표

  • 조합 테스트 축을 선정할 수 있다.
  • 2×2×2 = 8개 시나리오를 만들 수 있다.

4.2 이론(핵심 개념)

4.2.1 조합 테스트 필요성

  • 실제 사용자 환경은 "조합"으로 결함이 발생
  • 예: 비로그인 + 딥링크 + 네트워크 느림 = 로그인 후 목적지 복귀 실패

4.2.2 조합 축 Top 7 (자주 쓰는 것)

1. 사용자 상태: 비로그인 / 로그인 / 토큰만료

2. 라우팅: 일반진입 / 딥링크 / 알림진입

3. 네트워크: 정상 / 느림 / 끊김

4. UI 상태: 로딩 / 정상 / 빈 / 에러

5. 데이터: 없음 / 많음 / 특수문자

6. 권한: 허용 / 거부 / 설정 변경

7. 디바이스: 저용량 / 절전 / 저사양

4.2.3 조합 폭발 방지 규칙

  • 축은 3개만 선택
  • 각 축은 2값만 선택
  • → 2×2×2 = 8개 시나리오

4.3 실습(70분)

Step 1) 테스트 대상 기능 선정 (10분)

  • 예: "알림을 통한 상세 화면 진입"

Step 2) 조합 축 3개 선정 (10분)

  • 사용자 상태: 비로그인 vs 로그인
  • 라우팅: 알림진입 vs 일반진입
  • 네트워크: 느림 vs 정상

Step 3) 조합표 작성 (20분)

시나리오 사용자 라우팅 네트워크 결함 가설
1 비로그인 알림 느림 로그인 후 목적지 복귀 실패
2 비로그인 알림 정상 로그인 화면 표시
3 비로그인 일반 느림 로딩 상태 표시
4 비로그인 일반 정상 정상 진입
5 로그인 알림 느림 상세 로딩 지연
6 로그인 알림 정상 정상 진입
7 로그인 일반 느림 로딩 스피너 표시
8 로그인 일반 정상 정상 진입

Step 4) 시나리오 수행 (30분)

5) Day 9 — E2E(End-to-End) 시나리오 작성

5.1 학습 목표

  • E2E 시나리오를 작성할 수 있다.
  • 사용자 목적 달성 관점으로 테스트를 설계한다.

5.2 이론(핵심 개념)

5.2.1 E2E란?

  • 정의: 사용자가 "목적을 달성"하는 전체 흐름
  • 예시: 알림 → 상세 → 댓글 작성 → 완료

5.2.2 E2E vs 단위 테스트

구분 단위 테스트 E2E 테스트
범위 화면/기능 하나 여러 화면/기능 연결
목적 기능 동작 확인 사용자 목적 달성 확인
예시 로그인 버튼 클릭 로그인 → 홈 → 상세 → 액션

5.2.3 E2E 시나리오 작성 공식

제목: [사용자 목적] 달성 시나리오

사전 조건: (계정/데이터/권한)


절차:


1. 알림 클릭


2. 로그인 수행


3. 상세 화면 진입


4. 댓글 작성


5. 저장 확인


기대 결과:


    

  • 댓글이 저장됨
    • 

  • 알림 배지 제거
    • 

  • 상세 화면에서 댓글 표시
    •

```markdown
E2E-001: 알림을 통한 댓글 작성

사전 조건

  • 로그인 가능한 계정: test@example.com
  • 알림 수신 설정 ON
  • 네트워크 정상

절차

1. 푸시 알림 수신 (새 댓글 알림)

2. 알림 클릭

기대 결과

  • 로그인 후 상세 화면으로 자동 이동
  • 댓글이 정상 저장됨
  • 알림 배지 제거됨

수행 결과

  • [ ] Pass
  • [ ] Fail (사유: )

```
